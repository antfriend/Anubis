I wired the Mixing button to a new mixing editor screen, and added the screen rendering/update flow. The first-pass UI gives each model 4 mix slots with Enable, Source, Destination (Dest), Rate, and Offset. You can tap M1-M4 tabs to switch mix slots, tap source/destination to cycle channels, and use +/- for rate and offset. I also made the mix settings persist per model by storing them in the existing model data structure.

I treated this as a UI-first transmitter-style mixer editor, so the data model is there now, but it does not yet drive any actual output math. I assumed a simple 5-channel, 4-mix-slot setup as a good first shape.

|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|-|

I added the Mixing numpad. Tapping the numeric Rate or Offset value box now opens a compact pad above the footer with 0-9, -, and OK, plus a text box at the top that shows the value being entered. Tapping anywhere above the pad dismisses it without committing, and OK commits the typed value to the selected mix field.

I left the existing +/- tick buttons in place, so you now have both quick nudging and direct entry. The typed value is constrained to -100 through 100

I added the Aa shift key to the model-name keyboard, and it now toggles the alphabet rows between uppercase and lowercase while leaving numbers and special keys alone. I also set the startup default model name to Anubis for the primary slot if it’s blank, and changed new/reset mix defaults so Rate and Offset both start at 0.

Because the mix defaults changed, I bumped the mix storage version, so on first boot after flashing this build the saved mix slots will be reset once to pick up the new 0 / 0 defaults.

I added the press-and-hold rename behavior to the model list
The behavior is now:

tap a saved slot: selects that model like before
press and hold a saved slot for about 700 ms: opens the keyboard with that slot’s current name preloaded so you can edit it
tap an empty slot: still opens naming immediately

I implemented it with a small pending hold state so it doesn’t interfere with the normal release handling

Drive Type is now saved per model slot and restored when you switch models, just like reverse, trim, failsafe, and mixing.

One important nuance: the actual storage key is still the model slot (Model 1-Model 4), and the model name is the label attached to that slot. So when the user selects Model 1, all of that slot’s saved data now comes back together:

reverse switch states
trim values
failsafe switch states
mixing data
drive type

I also added boot-time sanitizing for old saved drive-type values so existing EEPROM data won’t leave the drive type in a bad state.

Symptom - top bar is not updating on any screen other than MAIN_SCREEN.

The issue was that the top bar only updates when uiNeedsRedraw is set, and the non-main screens don’t naturally trigger redraws very often. I added a small periodic redraw tick for non-main screens (250ms), so the RSSI bars and battery icon now refresh there too instead of only animating on the main screen.

The trim trail fix was redrawing the graph area but not the trim buttons afterward, so the + / - controls were getting painted over. I split the trim rendering into:

a graph-base redraw helper
a trim-button redraw helper
Now the dynamic trim update restores both before drawing the moving dot, so the trail stays gone and the buttons remain visible.

D-pad focus highlights now stay hidden until you actually use the D-pad on that screen, and they hide again as soon as you interact by touch or change screens. So the UI won’t open with a highlighted control unless navigation came from the D-pad.

I also added a Tank Control sub-screen under the Tank option in Drive Type. It gives you:

1 Stick: the right stick controls both sides with throttle on Y and turning on X

That setting is saved per model using the same per-model storage path as your other model settings, and the main-screen tank bars now follow that selected tank mode.

I updated the Reverse screen so enabled mixes now link reverse behavior between channels.
What it does now:

if an enabled mix connects channels, reversing one of those channels reverses the linked channel too
the Reverse screen visually flags linked channels with accent coloring and a *
linked channels also get an extra outline around their toggle, so the relationship is easier to spot

I implemented it as a general linked-channel resolver over the enabled mix graph, so it isn’t limited to only one specific pair.

I added a per-mix reverse-link switch to the Mixing screen. Each M1-M4 mix now has a Rev Link row with SEP and LINK, and that setting now directly controls the Reverse screen behavior. If a mix is LINK, reversing either channel in that mix will still flip the linked partner(s). If it’s SEP, that mix no longer participates in reverse linking or reverse-menu highlighting.

I kept the saved mix data layout compatible, so existing mixes should keep their current linked behavior by default instead of getting reset. I also shifted the Mixing layout a bit to make room for the new row and updated both touch and D-pad navigation to cover it cleanly.

I wired the battery meter to the board’s real battery ADC on IO9.
What changed:

battery voltage now comes from analogReadMilliVolts(IO9) instead of the old simulated value
I applied the board’s 200k/200k divider from the schematic, so the code converts the ADC reading back to actual battery voltage
if no battery is detected, the top-bar battery icon now shows an X and does not blink
the right panel now shows No Batt instead of a fake voltage
charging state is shown as an overlay on the battery icon and a + next to the voltage readout

One important limitation: this board does not appear to route a dedicated charge-status signal to the ESP32, so “charging” is inferred from the battery voltage rising over time, not from a true charger-status pin. That means the voltage reading is real, but charging detection is best-effort.
